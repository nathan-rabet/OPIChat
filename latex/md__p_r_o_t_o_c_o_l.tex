\hypertarget{md__p_r_o_t_o_c_o_l_autotoc_md10}{}\doxysection{Introduction}\label{md__p_r_o_t_o_c_o_l_autotoc_md10}
In OPIChat, clients can send messages to the server. We will call such messages {\ttfamily requests}.

These requests contain a command, indicating the action that needs to be performed by the server.

The server will reply with another message called either {\ttfamily response} or {\ttfamily error} depending on the completion of the request. It provides information about the status of the performed action.

A server might also send unsolicited responses to connected clients to inform them of an event (for example, notify that a client just received a message from another client). Such messages will be called {\ttfamily notifications}.

The OPIChat protocol is inspired by HTTP and IRC. We introduce you to a simplified version of these protocols so you do not lose time reading complex specifications.\hypertarget{md__p_r_o_t_o_c_o_l_autotoc_md11}{}\doxysection{Syntax}\label{md__p_r_o_t_o_c_o_l_autotoc_md11}
Requests, responses, notifications and errors in {\itshape OPIChat} all follow the same syntax.

They are composed of five main components\+: payload size, status code, command, command parameters and payload\+:


\begin{DoxyItemize}
\item The payload size is the number of bytes of the payload.
\item The status code is an integer indicating the type of the message or the possible errors. Every integer has its own meaning\+:
\begin{DoxyItemize}
\item A status code of 0 indicates that the message is a {\ttfamily request}.
\item A status code of 1 indicates that it is a successful {\ttfamily response}.
\item A status code of 2 indicates a notification from the server to a client (for example a message from a client to another).
\item A status code of 3 indicates that the message is a response to notify an error while processing a request, such message is called an {\ttfamily error}.
\end{DoxyItemize}
\item The command is a string identifying the action performed by the server.
\item The command parameters specify special requirements for a command (for example the username of a client you might want to send a message to). There can be any number of parameters in a message, each command as its own required parameters documented in the appropriate sections. Each parameter consists of a key/value pair with the following syntax\+: {\ttfamily key=value}. Parameters can be in any order in the message.
\end{DoxyItemize}

Each of theses elements are separated by a newline and there can be any number of parameters. The end of command parameters is marked by an empty line, which therefore represents the start of the payload.

For example the following is a request sent by a user named {\ttfamily ING1} asking the server to send the message {\ttfamily 2022} to the user {\ttfamily acu}\+:

\begin{quote}
4\$\textbackslash{} 0\$\textbackslash{} SEND-\/\+DM\$\textbackslash{} User=acu\$\textbackslash{} \$\textbackslash{} 2022 \end{quote}
In this example and all the others in this document, the {\ttfamily \$} character corresponds to a newline.

Such request will cause the server to send a response to the original client indicating the status of the operation. Upon successful completion, the response will look like this\+:

\begin{quote}
0\$\textbackslash{} 1\$\textbackslash{} SEND-\/\+DM\$\textbackslash{} User=acu\$\textbackslash{} \$ \end{quote}
While processing the request, the server will also generate a notification for user acu to notify them of a message\+:

\begin{quote}
4\$\textbackslash{} 2\$\textbackslash{} SEND-\/\+DM\$\textbackslash{} User=acu\$\textbackslash{} From=ING1\$\textbackslash{} \$\textbackslash{} 2022 \end{quote}
